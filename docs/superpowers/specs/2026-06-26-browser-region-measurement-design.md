# Browser-native per-region measurement — drop the "run locally" requirement

**Date:** 2026-06-26
**Status:** Approved design (pre-implementation), revised after adversarial spec review

## 1. Context & problem

FRAGRATE runs in two modes (`client/src/engine/mode.ts`):

- **Local mode** — the Node measurement server (`server/`) runs on the user's
  machine. Full report: per-region latency via raw TCP-connect timing
  (`server/src/pinger.ts`) and packet loss via server-side STUN/UDP
  (`server/src/stun.ts`).
- **Hosted mode** — static client served by a Cloudflare Worker
  (`worker/index.ts`). Today it measures only throughput, bufferbloat, and
  last-mile loss (browser WebRTC via Cloudflare TURN, `client/src/engine/loss.ts`).
  Per-region latency is unavailable, so the UI shows a **blocking "run locally"
  panel** (`client/src/App.tsx`):

  > "Real ping and packet loss to each game's regional servers need raw network
  > access a browser can't do. Run FRAGRATE locally to unlock the full per-game,
  > per-region report."

This asks every hosted visitor to install something to get the core value. We
want to remove that ask.

**Key insight:** the per-region "ping" targets *public HTTPS endpoints* —
`https://ec2.<region>.amazonaws.com:443` (`shared/regions.ts`, 13 regions). A
browser can reach `:443` via timed `fetch()`, from the user's *real* connection —
so per-region latency is recoverable in hosted mode with no local agent and no new
infrastructure.

## 2. Goals & non-goals

**Goals**
- Hosted (browser) mode produces the **full per-game, per-region report** with no
  local install.
- Stay "honest by design": every number labeled by how it was obtained.
- No new backend infrastructure; no added running cost.

**Non-goals**
- Matching raw-socket precision in hosted mode (exact UDP ping, true per-region
  UDP loss) — that remains local mode's job.
- Per-region *packet loss* in hosted mode (browsers can't see UDP loss over TCP).

## 3. Decisions (chosen)

1. **Browser-only, zero new infrastructure.**
2. **Keep local mode as an optional "precision mode."** Hosted is the default full
   experience; local stays for raw-socket ground truth, surfaced as a small
   optional note, not a blocking wall.

## 4. Design — the measurement core

### 4.1 Per-region latency → browser HTTPS-RTT probe

New module **`client/src/engine/regionPing.ts`**. For each region, time requests
to the same endpoint local mode uses (`https://ec2.<region>.amazonaws.com`).

**CRITICAL — do not follow redirects.** `GET https://ec2.<region>.amazonaws.com/`
returns **HTTP 301 → `https://aws.amazon.com/ec2`**. With the default
`redirect:'follow'`, every region would chase that redirect to the *same*
CDN-fronted host and all 13 latencies would collapse to one meaningless number.
The probe MUST use **`redirect:'manual'`** and time to promise resolution on the
resulting `opaqueredirect` response. The 301 is generated by the **in-region**
endpoint (verified: distinct per-region IPs and per-region timings), so a
non-following probe captures the true in-region RTT — and the 301 has no body, so
the timing is clean (no large-body download to confound it).

Probe per region:
1. **Warm-up request (discarded):** pays DNS + TCP + TLS once, parking the
   connection in the browser's keep-alive pool.
2. **N timed samples** on the reused connection:
   `fetch(url, { mode: 'cors', cache: 'no-store', redirect: 'manual', signal })`
   with a cache-buster query (`?_=<counter>`). **`mode` must be `'cors'` (the
   default), NOT `'no-cors'`** — the Fetch standard forbids `redirect:'manual'`
   with `no-cors` (no-cors requires `redirect:'follow'`), so `no-cors` + `manual`
   throws synchronously in the browser. `cors` + `manual` resolves with an
   `opaqueredirect` (CORS headers not needed; the body is unreadable either way).
   A simple GET with no custom headers triggers no CORS preflight.
   Each is ~1 round-trip. Compute
   **min** (cleanest RTT), **median** (stable), and **jitter = mean of
   `|sample[i] − sample[i-1]|` over consecutive samples** (IPDV — identical
   formula to `server/src/pinger.ts:116-119`, so hosted and local jitter are
   directly comparable).
3. **Output shape = `ProbeStats` (`shared/protocol.ts`)** — NOT `PingStats`. The
   store, `RegionSelector`, and grading all consume `ProbeStats`, which has an
   `id` field (the region id) and no `max`/`error`. Each result sets
   `id` = region id, `host`/`port` from `REGIONS`, `samples`, `received` (count of
   successful samples), `lossPct`, `min`/`avg`/`median`/`jitter`. Write each via
   `store.putRegion(stats)` (keyed on `stats.id`), exactly as the server's
   transform does (`server/src/ws/session.ts:75-87`).
4. **All 13 regions run in parallel** (samples sequential within a region) so
   game/region switching recomputes instantly from already-measured data.

**Why this technique:** the only browser-native option against these existing
public endpoints. WebSocket isn't served by them; precise Resource-Timing needs a
`Timing-Allow-Origin` header we can't set without our own infra (ruled out). The
result is *the user's* path to that metro, measured one layer up — labeled
**"HTTPS RTT"** (local keeps **"TCP handshake"**). No calibration/offset is applied
(would violate the honesty ethos).

**Honesty caveats (must be reflected in labels, §9):** HTTPS RTT reads **somewhat
above** raw UDP ping (TLS/HTTP/app-layer, plus any per-host server processing on
the 301); min-of-N sampling and the distinct label cover this, but absolute hosted
numbers are **not directly comparable** to local TCP-handshake numbers. Likewise
hosted **jitter is HTTPS/app-layer-derived** and not the same physical quantity as
local handshake jitter; it still feeds the same grading jitter hard-cap
(`shared/grading.ts:111`) — this is accepted, but the jitter source is disclosed.

### 4.2 Packet loss → connection-wide last-mile (reuse existing probe)

Browsers can't measure per-region UDP loss — TCP retransmits hide it. The existing
`measureBrowserLoss()` (WebRTC forced through Cloudflare TURN,
`iceTransportPolicy:'relay'`) measures **last-mile UDP loss** (browser → nearest CF
edge → browser). Last-mile is where consumer packet loss originates, so it applies
to every region.

Decision: present it as **one connection-wide loss number** that feeds every
per-game verdict. This is the **idle** WebRTC measurement (`lossLoaded` stays
`null` in hosted; grading already renormalizes weights when loaded loss is
absent). Keeps the existing TURN dependency via `/api/turn`.

- **Loss method/label:** the hosted report's `LossSummary.method` MUST be
  **`'webrtc'`** (as `assembleHostedReport` already sets), never `'stun-udp'`
  (which `assembleReport` hardcodes — see §6). UI label: **"last-mile UDP loss
  (your connection)."**
- **Hard-cap disclosure (honesty):** `grading.ts:109` caps a verdict to rank C
  whenever `loss_pct > bands.lossPct.bad`, and those edges are small (1.5–2.5%).
  Because hosted loss is one connection-wide number, a single high reading caps
  **every game and every region identically**, even regions whose latency is
  excellent. This global cap is **intentional and accepted** (your last mile does
  bound every path), but it MUST be disclosed in the verdict UI — e.g. "limited by
  your connection (last-mile loss), not this region" — so a uniformly pessimistic
  result is never mistaken for a per-region judgment.

### 4.3 Unchanged metrics

Throughput and bufferbloat are already browser-based (Cloudflare endpoints) in
hosted mode. No change.

## 5. UX changes (`client/src/App.tsx`, `client/src/i18n.ts`)

- **Remove the blocking "run locally" panel** (`App.tsx:150-165`). Drop the
  `s.mode === 'hosted'` fork so **both modes render the same `RegionSelector`**,
  with `allowed={gameRegions(s.selectedGameId)}` and `selected={s.selectedRegion}`,
  `onSelect={onPickRegion}` — populated with browser-measured per-region latency.
- **Local mode → optional footnote.** Reword the run-locally strings into a
  subtle, non-blocking note: *"Measured in your browser (HTTPS RTT + last-mile
  loss). Want raw-socket precision — exact UDP ping and true per-region loss? Run
  FRAGRATE locally."*
- **Rename badge** `"Hosted demo"` → **`"Hosted"`**.
- **Mode-aware "How to read this" note.** `noteBody` (`i18n.ts:186-191`, rendered
  in both modes at `App.tsx:167-169`) is currently hardcoded to local-mode claims
  ("TCP-handshake time… STUN/UDP probes to public STUN servers"). It MUST branch
  on mode — hosted variant: latency = HTTPS RTT to each region's public endpoint
  (somewhat above raw UDP ping); loss = last-mile UDP loss via WebRTC/Cloudflare
  TURN (connection-wide, not per region).
- **Surface the graded region.** Grading picks the best in-game region
  (`bestRegion`). Hosted must show **which** region the verdict was graded on
  (`report.region` already exists) and indicate when regions were unreachable, so a
  "Playable" verdict can't silently rest on a non-representative best region while
  others failed.
- **Per-metric honesty labels** per §9. New/changed strings added in **EN and ES**.
  Dead strings (`internetRtt`, `hostedApprox`) are removed or repurposed;
  `latencyTcp` stays local-only.
- **No new phase value.** The first phase id is **`'regions'`** (UI short-label
  "Ping"); hosted already drives it (for a single idle sample) and will now run
  `measureAllRegions()`. `PhaseStepper` is unchanged.

## 6. Components & files

**New**
- `client/src/engine/regionPing.ts` — pure stats core (min/median/jitter, lossPct
  from samples — unit-testable) + a thin timing layer behind an injectable
  "time one request" function; exports `measureAllRegions(onProgress)` returning
  `Record<Region, ProbeStats>` (or writing each via `store.putRegion`).

**Changed**
- `client/src/engine/orchestrator.ts` — **central change, not additive:**
  - `runHosted` (`:341-406`): in the `'regions'` phase, call `measureAllRegions()`,
    populate `store.regions`, compute the chosen region via
    `gameRegions(gameId)` + `bestRegion` (mirroring `runLocal`), **set
    `store.selectedRegion`** (runHosted never sets it today), and assemble the
    report via the **region-aware `assembleReport`** path — NOT the region-less
    `assembleHostedReport` (which hardcodes `region:'NA-East'`, `regions:[]`, and a
    synthetic `id:'internet'` ping). Feed connection-wide loss from `m.lossIdle`
    with `LossSummary.method='webrtc'`.
  - `recompute` (`:160-183`): the hosted branch currently short-circuits ("No
    regions in hosted mode") and ignores `opts.region`. It MUST become
    region-aware — re-pick/re-grade against `store.regions` like the local branch
    and update `store.selectedRegion` — or game/region clicks won't recompute.
  - `assembleReport` hardcodes `loss.method='stun-udp'`; parameterize the loss
    source/method (or branch) so the hosted path emits `method:'webrtc'`.
    `assembleHostedReport` is either retired or kept only for the
    all-regions-failed fallback (§8).
- `client/src/App.tsx` — drop the hosted fork; render `RegionSelector` in both
  modes (§5); mode-aware note; badge rename; show graded region + unreachable
  indication.
- `client/src/i18n.ts` — mode-aware `noteBody` (EN/ES); new labels ("HTTPS RTT",
  "last-mile UDP loss (your connection)", "Hosted"); remove/repurpose dead strings.
- `client/package.json` — add Vitest devDep + `test` script (§10).

**Unchanged**
- `server/` (local raw measurements), `client/src/engine/mode.ts`,
  `worker/index.ts`, `shared/protocol.ts` (`LossSummary.method` already allows
  `'webrtc'`; no type change needed).

**Not a change: CSP.** There is **no Content-Security-Policy anywhere** in the repo
(verified: no `<meta>` in `client/index.html`, no header in `worker/index.ts`, no
`_headers`, no `headers` in `wrangler.jsonc`). So nothing blocks the AWS hosts
today, and `wrangler.jsonc` needs no edit. This is a **verification step**, not a
work item: if a CSP is ever added, `connect-src` must include
`https://ec2.*.amazonaws.com`.

## 7. Data flow — hosted RUN TEST

1. `detectMode()` → `hosted` (worker 404s `/api/health`).
2. RUN TEST → orchestrator phases:
   - **`'regions'` (UI "Ping"):** `measureAllRegions()` → per-region HTTPS RTT
     (parallel, warm-up + samples, `redirect:'manual'`) → `store.regions[id]`
     `ProbeStats`. Choose region via `gameRegions` + `bestRegion`; set
     `store.selectedRegion`.
   - **`'loss'`:** `measureBrowserLoss()` → connection-wide last-mile loss
     (`method:'webrtc'`, idle only).
   - **download / upload / bufferbloat:** Cloudflare (unchanged).
   - **scoring:** `assembleReport` grades within `gameRegions` using per-region
     latency + connection loss + throughput → per-game verdicts; recompute on
     game/region switch is instant (via the fixed `recompute`).
3. UI renders the region map + verdicts + gauges with hosted labels, showing the
   graded region.

## 8. Error handling & graceful degradation

- **One region fails** (timeout/blocked) → write a `ProbeStats` for it with
  `received:0`, `median:null` (NOT an absent key), so `RegionSelector`'s
  `received > 0` check renders "—". Others still render.
- **All regions fail** (corporate proxy, etc.) → fall back to the optional
  footnote surfaced prominently ("couldn't reach region endpoints from your
  browser — run locally for per-region ping"); keep the `assembleHostedReport`
  region-less path for this case. No crash.
- **Loss** already degrades to `available:false` when `/api/turn` is non-OK
  (`loss.ts:48`); grading renormalizes (drops the loss term) rather than scoring
  unmeasured loss as 0%. Unchanged.

## 9. Honesty labels (summary)

| Metric | Local mode | Hosted mode |
|---|---|---|
| Per-region latency | "TCP handshake" | "HTTPS RTT" (somewhat above raw ping; not directly comparable) |
| Jitter | handshake-derived | HTTPS/app-layer-derived (disclosed) |
| Packet loss | server STUN/UDP (`method:'stun-udp'`) | last-mile UDP loss, connection-wide (`method:'webrtc'`); high reading caps all regions — labeled as connection-limited |
| Throughput / bufferbloat | Cloudflare | Cloudflare |
| "How to read this" note | local wording | hosted wording (mode-aware) |

## 10. Testing

- **Unit (new, deterministic):** the pure stats core (min/median/jitter matching
  `pinger.ts`; `lossPct` from sent/received) with golden cases; region-result
  mapping with a **mocked timing function** (no network) — including the
  failed-region → `received:0, median:null` representation.
- **Probe-correctness assertion:** a test/assertion that the resolved response is
  `type === 'opaqueredirect'` (or, manually, that distinct regions yield distinct
  latencies) — guarding against the redirect-follow regression.
- **Test infra:** the repo has no test runner today. Add **Vitest** as a
  devDependency in the **client** workspace, a `vitest.config.ts` (no jsdom — pure
  functions), a `test` script in `client/package.json` (optionally a root `test`
  that filters to client), test file `client/src/engine/regionPing.test.ts`.
  (Fallback: manual verification only, if test infra is unwanted.)
- **Manual:** deploy hosted; confirm the region map populates with **distinct,
  plausible** per-region latencies (not all identical — the redirect check),
  verdicts compute, game/region switching recomputes instantly, the graded region
  is shown, and an unreachable region renders "—". Regression: local mode still
  uses the Node TCP pinger.

## 11. Out of scope / future

- **True per-region UDP loss + precise per-region latency** via lightweight
  regional responders. The `regionPing` injectable-timing seam and the
  connection-wide loss boundary leave a clean place to add this later.

## 12. Implementation parameters (pin these in the plan)

- **Samples:** warm-up (1, discarded) + **7** timed samples per region.
- **Per-request timeout:** 2000 ms (mirrors `pinger.ts` default) via `AbortSignal`.
- **Cache-buster:** `?_=<incrementing counter>` (avoids `Math.random`/`Date.now`
  pitfalls; any monotonic counter is fine).
- **Parallelism:** all 13 regions concurrent (13 distinct hosts — within browser
  caps); samples sequential within a region.
- **Failed sample vs failed region:** a sample that times out/errors is dropped
  (counts against `received`); a region with 0 successful samples →
  `received:0, median:null`.
- **Expected `'regions'` phase budget:** ~1 warm-up + 7 samples per region, run in
  parallel; bounded by the slowest region (~a few seconds worst case).

## 13. Risks & open questions

- **HTTPS-RTT magnitude:** above raw UDP ping (TLS/app-layer + 301 server
  processing); mitigated by min-of-N and honest labeling; absolute numbers not
  comparable to local.
- **`redirect:'manual'` + `cors`:** yields an `opaqueredirect` response that
  resolves at the in-region 301 (no body) — the intended clean ~1-RTT timing.
  (`no-cors` + `manual` is invalid and throws — see step 2.) Verified in a real
  headless browser across 7 regions: all `opaqueredirect`, distinct per-region
  RTTs (NA ~50ms → Asia-South ~270ms), no errors.
- **Connection reuse:** HTTP keep-alive is keyed on origin (host:port), so the
  cache-buster query/path does not break reuse; warm samples stay ~1 RTT. A
  dropped connection appears as an outlier, filtered by min/median.
